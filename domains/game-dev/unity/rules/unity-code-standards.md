# unity 项目全局规则

这些规则适用于任何 unity 项目。

## unity 项目通用代码规范

### 命名规范

#### 大小写通用规则

- PascalCase（大驼峰）：类名、接口名、方法名、公共属性、公共字段、枚举类型与枚举值、事件、委托类型、常量。
- camelCase（小驼峰）：局部变量、方法入参。
- 接口名统一以大写 I 开头，例如 IDamageable、IInteractable。
- 特性类统一以 Attribute 结尾，例如 ReadOnlyAttribute。

#### 字段命名约定

- 私有实例字段：行业通用格式为 _camelCase（下划线开头 + 小驼峰），例如 _playerHealth、_moveSpeed，便于快速区分成员变量与局部变量。
- 私有静态字段：.NET 官方约定为 s_camelCase 前缀。
- 序列化私有字段（加 [SerializeField]）遵循私有字段命名规则，不因暴露到 Inspector 改用大驼峰。

### 代码格式与结构规范

#### 大括号风格

- 采用 C# 官方标准的 Allman 风格，每个大括号单独占一行，不跟在语句末尾。

#### 缩进与空格

- 使用 4 个空格缩进，不使用制表符（Tab）；运算符前后加空格，提升可读性。

#### 成员排序规则

- 类内成员按固定顺序排列，便于快速定位：
构造函数 → 属性 → Unity 生命周期函数（如果有） → 公共方法 → 私有方法 → 事件 → 嵌套类型 → 实例字段 → 静态字段 → 常量
- MonoBehaviour 的生命周期函数必须按引擎执行顺序集中排列，不要零散穿插在业务方法中：
Awake → OnEnable → Start → FixedUpdate → Update → LateUpdate → OnDisable → OnDestroy

#### 其他格式约定

- 显式声明访问修饰符，即使是 private 也不省略，避免歧义。
- 优先使用卫语句（尽早 return），减少深层嵌套的 if-else，降低代码圈复杂度。
- 一个文件对应一个主类，文件名与类名完全一致。

### 注释与文档规范

#### 注释范围

- 所有对外的公共 API（类、方法、属性）必须添加 XML 文档注释，说明功能、参数、返回值。
- 内部私有方法按需添加，普通注释只说明「设计原因、注意事项、边界条件」，不重复代码本身能直接读懂的逻辑。

#### 标记注释

使用统一前缀标记待办与问题，便于全局检索：
- // TODO: 待实现的功能
- // FIXME: 已知需要修复的问题
- // NOTE: 特别说明的设计细节

#### Unity 专属注释

- Inspector 中展示的序列化字段，使用 [Tooltip("说明")] 特性补充解释，替代代码内的冗余注释。

#### 区域折叠

- 合理使用 #region 折叠大段同类代码（如所有生命周期函数、所有公共方法），不要过度折叠破坏阅读连贯性。

### Unity 脚本与生命周期专属规范

#### 删除空生命周期函数

- 空的 Update、Start 等方法即使没有代码，引擎也会通过 native 调用产生固定开销，必须删除。

#### 生命周期职责分离

- Awake：仅做自身初始化（获取自身组件、初始化内部数据），此时其他对象的 Awake 可能还未执行。
- Start：处理依赖其他对象的逻辑，此时场景内所有对象的 Awake 均已执行完毕。

#### 禁止高频查找

- 不在 Update/FixedUpdate 中调用 GetComponent、FindObjectOfType 等查找方法，统一在 Awake/Start 中缓存为成员字段。

#### 协程规范

- 用 Coroutine 句柄启动和停止协程（接收 StartCoroutine 的返回值），不使用字符串名称启动，避免拼写错误且性能更优；协程必须配套停止逻辑，防止对象销毁后泄漏。

#### 慎用反射类 API

- 避免使用 SendMessage、BroadcastMessage，反射开销大且类型不安全，改用接口或事件替代。

#### 单例规范

- MonoBehaviour 单例必须处理场景切换、重复创建、销毁顺序等问题；优先使用依赖注入（如 Zenject、VContainer）替代单例，降低模块耦合。

### 代码质量与可维护性规范

#### 不可变修饰

- 运行期不会改变的变量，使用 const（编译期常量）或 readonly（运行期常量）修饰，提升安全性与编译器优化空间。

#### 消除魔术数字

- 代码中不直接出现无意义的数字常量，定义为 const 或枚举，提升可维护性。

#### 方法单一职责

- 一个方法只做一件事，避免写超长方法；超过一屏的方法建议拆分为多个子方法。

#### 减少静态状态

- 慎用静态变量存储跨场景状态，容易产生状态残留导致的隐蔽 Bug；跨场景数据使用专门的全局管理类或持久化方案。

#### var 使用原则

- 当代码右侧能清晰看出类型时可使用 var；类型不明确时显式声明类型，不牺牲可读性。
