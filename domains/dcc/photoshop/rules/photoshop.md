# Photoshop DCC Development Rules

Rules for AI-assisted Photoshop plugin and tool development with ExtendScript (.jsx), UXP (.psjs + manifest.json), and C++ SDK (.cpp/.h). Follow these in every Photoshop scripting project. Target versions: Photoshop 2022–2026 (v23–v27). **MUST NOT: Fabricate API signatures. All ExtendScript, UXP, and C++ SDK API references must be verifiable against Adobe documentation.**

## Project Structure

- **ExtendScript projects**: Place scripts in `~/Adobe Scripts/` or a version-controlled repository with symlinks. Use `scripts/` for entry-point `.jsx` files, `includes/` for shared `.jsxinc` modules, and `docs/` for usage documentation. Entry scripts must contain `#target photoshop` at line 1 for cross-platform file-launch compatibility. Use `.jsxinc` extension for include files — Photoshop's ExtendScript engine distinguishes these from standalone `.jsx` scripts and they cannot be executed directly.
- **UXP projects**: Root `manifest.json` with `"manifestVersion"` 4+ and `"host": {"app": "PS", "minVersion": "23.0.0"}`. Structure source under `src/`, built output under `dist/`, and static assets under `public/`. Entry point defined by `"main": "index.html"`. Include `"entrypoints"` array for command and panel definitions. Panel plugins require icon PNG assets at 23px and 46px sizes for both dark and light themes. Plugins destined for the Adobe Marketplace must ship a single `host` definition, not an array.
- **C++ SDK projects**: Opaque to AI scaffolding — do NOT attempt to generate project files, build configurations, or platform toolchain setup. Recognize PiPL `.r` resource files that define plugin type (filter `.8bf`, automation, format, import, export) and entry functions (`PluginMain` for filters, `AutoPluginMain` for automation). Reference: source in `source/`, SDK headers in `include/`, platform projects in `projects/win/` or `projects/mac/`. Hybrid UXP+C++ plugins register `PSDLLMain()` as the Photosop-side entry; pair with a UXP `manifest.json` containing the matching component ID.
- Separate Photoshop-specific logic from framework-level concerns: ExtendScript UI in `ScriptUI` resource strings, UXP UI in HTML/CSS with Spectrum Web Components design tokens, C++ UI via Photoshop's `PIDialog` suite.

## Coding Conventions

- **ExtendScript**: Target ES3 engine only — `var` for all variables, no `let`, `const`, arrow functions, `Promise`, `async/await`, template literals, or `for...of`. Use `#target photoshop` directive at top of entry scripts. Group shared helpers in `.jsxinc` include files accessed via `#include "path/to/helpers.jsxinc"`. Global `$` object provides ExtendScript extensions: `$.writeln()` for logging, `$.level` for interpreter-version targeting, `$.global` for cross-script state. Enclose all script logic in an IIFE to prevent global-scope pollution: `(function() { ... })()`. Use function declarations over function expressions for consistent hoisting behavior under ES3.
- **UXP**: Every `.psjs` file must begin with `"use strict"`. Import Photoshop DOM via `const app = require('photoshop').app` and core utilities via `require('photoshop').core`. All document-modifying operations require the modal execution wrapper: `await require('photoshop').core.executeAsModal(async (executionContext) => { ... })`. Use `batchPlay()` with action descriptor arrays only when the DOM API lacks the required feature — check the Photoshop API reference before falling back to `batchPlay`. Manifest v5+ requires explicit `requiredPermissions` block; undeclared permissions are denied at install time with no runtime fallback. Never use `require('uxp').storage.localFileSystem.getFolder()` without the `"localFileSystem"` permission.
- **C++ SDK**: Every suite access follows `sSPBasic->AcquireSuite(kSuite, kVersion, (const void**)&suitePtr)` and must check `kSPNoError` return. All functions return `SPErr`. `PI*` prefix denotes Photoshop-specific types (`PIActionDescriptor`, `PIActionReference`, `PIUXPSuite`). Memory for descriptors and references obtained via suite `Make()` calls must be explicitly released via corresponding `Free()` calls — no garbage collection. Use `PSActionControlProcs::StringIDToTypeID()` and `TypeIDToStringID()` for runtime string-to-descriptor-key conversion; never hardcode four-character codes.

## API Usage Patterns

- **ExtendScript DOM**: Always guard `app.activeDocument` with `app.documents.length > 0` — accessing `.activeDocument` without documents open throws a runtime error. Access layers via `app.activeDocument.layers`; iterate with `for (var i = 0; i < layers.length; i++)`. Use `app.doScript()` with action name and descriptor to execute recorded Actions panel operations. Add undo steps via `app.activeDocument.suspendHistory("undoName", "actionString")` for user-facing tools. For filter operations, use `app.activeDocument.activeLayer.applyStyle()` and `applyChannelMixer()` as documented in the Photoshop JavaScript Reference.
- **UXP DOM**: Access document via `app.activeDocument` — returns `undefined` when no document is open, so check before use. Layer creation uses `await doc.createLayer({name, opacity, blendMode})` inside `executeAsModal`. Save documents with `await doc.save()` or `await doc.saveAs()` using `require('uxp').storage` token APIs for file selection. For APIs not in the DOM, use `batchPlay()` with action descriptors constructed via `require('photoshop').action`: `charIDToTypeID()`, `stringIDToTypeID()`, descriptor `putString()`, `putInteger()`, reference `putProperty()`. Use `app.showAlert()` for user-facing messages instead of `window.alert()` which is not available.
- **C++ SDK**: Acquire suites via `SPBasicSuite` passed to entry function. Use `PSActionControlProcs::Play()` to dispatch events, `::Get()` to read properties. Construct references with `PIActionReference` suite, descriptors with `PIActionDescriptor` suite. For UXP interop, use `PsUXPSuite1::SendUXPMessage()` passing the UXP plugin's `manifest.json` `id` string. Hybrid plugins register `PSDLLMain()` entry for Photoshop to invoke on load. Use `SPAccessRef` for file picker integration with the host application.
- Null-safety is consistent across surfaces: empty collections (`layerSets`, `channels`, `pathItems`) return zero-length arrays/collections, not `null`. Always check `.length` or iterate safely.

## Performance

- **ExtendScript**: Avoid per-layer API calls inside loops — batch operations using `app.doScript()` to wrap the full workflow as a single action execution. Place expensive file I/O inside `#include`-loaded modules that are parsed once at initialization. ExtendScript is single-threaded and synchronous; any long-running script blocks Photoshop UI entirely. For batch processing of many files, use `app.open()` / `app.activeDocument.close(SaveOptions.DONOTSAVECHANGES)` patterns with careful resource cleanup.
- **UXP**: `batchPlay()` accepts arrays of action descriptors — group multiple Photoshop operations into a single `batchPlay()` call for optimal throughput. DOM API methods (`doc.layers`, `doc.createLayer()`) use internal batching where possible, so prefer them over `batchPlay` when the feature is supported. Offload non-Photoshop heavy computation to web workers; UXP panels run a Chromium-based runtime that supports workers for CPU-bound tasks. Avoid synchronous `batchPlay()` with `synchronousExecution: true` — it blocks both the UXP panel and Photoshop UI thread.
- **C++ SDK**: Request large memory allocations through Photoshop's `PIBufferSuite` (`New64`, `GetSize64`) rather than direct `malloc`/`new` — Photoshop's memory manager handles fragmentation across plugins. Use `BufferProcs::AllocateBufferProc64` for pixel-sized scratch buffers. Format plugins should set the `SupportsBackgroundSave` PiPL property to enable asynchronous document saves without blocking the main thread. Use `FilterRecord::bufferSpace64` and `maxSpace64` for 64-bit buffer requests in modern Photoshop.

## Common Pitfalls

- **ExtendScript**: `app.documents[0]` throws an exception when no documents are open — always guard with `app.documents.length > 0`. ScriptUI dialogs block the main thread and freeze Photoshop; never run document-processing loops inside `onClick` handlers. `$.setenv()` and `$.getenv()` state persists across script invocations in the same Photoshop session — clear temporary keys after use. `executeAction` with mismatched descriptor keys does not produce an error — validate descriptor strings against the Photoshop Scripting Listener output.
- **UXP**: Forgetting `executeAsModal` on document-modifying code causes silent failure — no error is thrown, the operation simply does nothing. Manifest `apiVersion` defaults shift: `minVersion < 23.0` → `apiVersion: 1` (deprecated synchronous model), `minVersion >= 23.0` → `apiVersion: 2` (required modal model). Panel plugins cannot access filesystem or network APIs unless corresponding permissions are declared in `requiredPermissions`. Upgrading from manifest v4 to v5 without adding `requiredPermissions` breaks all previously-working filesystem/network access.
- **C++ SDK**: `PIActionDescriptor` and `PIActionReference` leaked after `Make()` → memory grows with every call until Photoshop is restarted. Path separator differs by platform: `\\` on Windows, `/` on macOS — never hardcode a single separator. `SPPluginRef` and `SPBasicSuite` pointers become invalid after Photoshop restart; never cache them across sessions. UI-displaying suites (`PIDialog`) must only be called from the main thread — calling from background save/export handlers causes undefined behavior.

## Testing

- **ExtendScript**: Use ExtendScript Debugger (bundled with ESTK or via VS Code ExtendScript extension) for step-through debugging and `$.writeln()` console output. Wrap top-level script in try-catch with `$.global["catch"] = function(e) { $.writeln(e) }` for uncaught exception reporting during development. Run scripts via `File > Scripts > Browse` in Photoshop or command-line `photoshop.exe script.jsx`. Use the Photoshop Scripting Listener (plug-in) to record actions as script code for reference during development.
- **UXP**: Load plugins via UXP Developer Tool (UDT) connected to a running Photoshop instance. `console.log()` output appears in UDT's Developer Console panel. Inspect DOM state via Photoshop menu `Plugins > Development > UXP Developer Tool`. Validate manifest before packaging with `UXP Developer Tool > Validate`. Test across multiple Photoshop versions by varying the connected instance. For `batchPlay()` debugging, log descriptor JSON before execution to verify key/value structure.
- **C++ SDK**: Loading a C++ plugin requires copying the built `.8bf` (or `.plugin` on macOS) to Photoshop's `Plug-ins/` directory and restarting the application. Report errors via `FilterRecord::errorString` buffer. No headless testing framework exists — validate filter output through visual comparison in Photoshop or exported file checksums.
- Cross-surface testing: ExtendScript and UXP share the same Photoshop document model; test UXP plugins against equivalent ExtendScript scripts to verify behavioral consistency. C++ hybrid plugins must be tested with their corresponding UXP frontend loaded simultaneously.

## Migration & Interoperability

- **ExtendScript to UXP**: The `executeAction` / `executeActionGet` API maps to UXP's `batchPlay()` — use the ExtendScript `ps-es-to-uxp` logger utility to auto-convert action descriptor calls. ExtendScript is synchronous; all migrated UXP code must use `executeAsModal` wrappers around formerly-synchronous document operations.
- **UXP to C++ hybrid**: UXP plugins communicate to C++ side via `require('photoshop').messaging.sendSDKPluginMessage(componentId, messageContent)`. C++ side receives messages through `PsUXPSuite1::AddUXPMessageListener()` with a `PIUXPMessageNotifier` callback. Message payloads are `PIActionDescriptor` objects built with the PICA suite.
- **Version bridging**: When targeting both older (ExtendScript) and modern (UXP v5+) surfaces, maintain separate entry scripts that share the same business logic in `.jsxinc` files parsed by both environments. UXP's DOM coverage expands version by version — always verify API availability against the target `minVersion` before assuming a DOM method exists.
