## 设计系统与组件开发工作流

### 1\. 有明确视觉参考

- **静态参考**（截图、原型、Figma 导出）：加载 `image-to-code-skill.md`，先提取参考中的设计令牌、布局、文案、间距、交互状态、响应式规则写入 `DESIGN.md`，再开发可复用组件。

- **线上站点参考**：加载 `clone-from-url.md`，通过真实浏览器提取运行时样式写入 `DESIGN.md`，再复刻实现。

- 最终验收：像素级视觉对比校验，确保实现精准且组件可复用。

### 2\. 全新项目（无参考）

设计调研是必经步骤，需并行执行所有调研方向，并在 `DESIGN.md` 中记录调研日志：

- 从内置参考库筛选 1 个 Layer A 风格技能 \+ 1 个 Layer B 品牌设计系统

- 按规范抓取真实产品界面，提炼布局规律

- 生成 2\-3 版概念稿，选定最优版作为视觉基准

所有调研结果整合进 `DESIGN.md`，提炼为项目专属的设计令牌与组件规范，再进入开发。

### 3\. 已有项目且含 \[DESIGN\.md/\]\(DESIGN\.md/\) 组件体系

先读取现有设计系统并严格遵循；仅当需求新增令牌、组件、交互状态时，先更新 `DESIGN.md` 再开发。

### 4\. 已有 UI 但无 \[DESIGN\.md\]\(DESIGN\.md\)、无可复用组件

暂停开发，询问用户：是保留现有外观做就近样式修改，还是先提炼完整 `DESIGN.md` 与可复用组件再继续。

> 凡创建 / 更新 `DESIGN.md` 的工作，默认加载 designpowers 相关规则，将用户画像、无障碍、评审标准、技术债等内容纳入设计系统。

## 四大规则集说明

### 规则集 1 — design（设计体系）

存放于 `references/design/`，包含架构规范、12 种风格技能（Layer A：实现方法）、70\+ 品牌设计系统（Layer B：视觉参考）。

- **Layer 0 架构**：`design-system-architecture.md` 定义 `DESIGN.md` 的标准结构。

- **Layer A 风格技能**（最多选 1 种，风格互斥）：

    - 通用实用风：工具类后台、仪表盘默认选择

    - 高级营销风：Awwwards 级、滚动动效落地页

    - 极简风：Notion/Linear 风格、编辑感界面

    - 粗野主义：瑞士风格、实验性设计

    - 精致高端风：奢华感、SaaS 产品界面

    - 存量改版：优化现有 UI，先审计再改版

    - 图转代码：先出设计稿再编写代码

    - 纯图像生成：仅输出原型图，不写代码

- **Layer B 品牌参考**：可直接对标 linear\.app、stripe、vercel、apple、notion 等 70\+ 品牌的设计系统，提取令牌应用到自有项目。

- **React 开发工具**：提供 react\-grab /react\-scan 等工具的接入方案。

### 规则集 2 — perfection（性能与质量）

存放于 `references/perfection/`，用于前端代码审计与性能优化。

- 核心准则：仅基于真实浏览器跑分，追求 Lighthouse 全项满分，从架构层面优化，不牺牲交互与视觉体验换分数。

- React 专项：提供 Playwright \+ Lighthouse \+ React 性能检测的完整方案，杜绝无意义重渲染。

### 规则集 3 — ui\-ux\-db（样式数据库）

存放于 `references/ui-ux-db/`，可检索的设计规范数据库，支持 CLI 查询。

- 支持查询：配色、字体、落地页结构、图表、UX 规范、技术栈最佳实践。

- 支持技术栈：HTML\+Tailwind、React、Next\.js、Vue、Svelte、shadcn 等。

### 规则集 4 — designpowers（设计运营层）

存放于 `references/designpowers/`，为设计工作提供运营层支撑。

- 覆盖：用户画像、无障碍与认知约束、设计评审、技术债管理、交付规范、模拟用户测试、动效指引。

- 需先将相关要求提炼进 `DESIGN.md`，再作为开发依据。

## 通用核心准则（所有规则集强制遵循）

1. **无设计系统，不做 UI 开发**：所有颜色、字号、间距必须能追溯到 `DESIGN.md` 中的设计令牌。

2. **明确参考即合同**：有截图 / 原型参考时，必须严格匹配像素、文案、组件结构与响应式逻辑，用户同意偏差除外。

3. **不牺牲体验换性能**：不得为了 Lighthouse 分数删减动效、隐藏内容、简化交互，必须视觉与性能同时达标。

4. **禁止用 emoji 当图标**：仅使用 SVG 图标库（Lucide、Heroicons 等）。

5. **仅使用 GPU 友好动画**：只允许动画 `transform`、`opacity`、`filter` 属性，禁止动画布局属性。

6. **拒绝无意义动效**：所有动画 / 悬停效果必须对应真实交互或状态变化，纯装饰性动效禁止添加。

7. **以视觉 QA 为完成标准**：通过真实浏览器多尺寸、多状态的像素级校验，才算交付完成。
