# Git 大师（git\-master）

**适用场景**：所有涉及提交操作、Git 历史排查的任务必须使用本技能。覆盖原子提交、暂存、提交信息规范、变基、压缩提交、fixup/autosquash、代码追溯、二分查找、引用日志、`git log -S/-G` 等能力，可解决 “谁编写了这段代码”“内容何时新增” 这类问题。普通代码编辑无需启用，仅当用户明确提出 Git 相关操作时使用。

## 操作分类

先判定请求所属类型：

- **提交（COMMIT）**：暂存并提交本地变更

- **变基（REBASE）**：变基、压缩、fixup、自动压缩、重排、拆分等分支历史改写操作

- **历史查询（HISTORY）**：查询变更的时间、位置、作者、原因，定位对应提交

- **状态检查（STATUS）**：查看分支、差异、工作区状态，不修改任何内容

> 安全原则：除非用户明确要求，否则不得执行提交、变基、推送、强制推送、重置、弹出暂存、删除等写操作。仅查询类需求输出结果即可，不额外执行修改。

## 基础信息采集

操作前先获取仓库真实状态，基于事实判断：

```bash
git status --short          # 工作区状态
git diff --stat             # 未暂存变更统计
git diff --staged --stat    # 已暂存变更统计
git branch --show-current   # 当前分支
git log -30 --oneline       # 最近30条提交摘要
git rev-parse --abbrev-ref @{upstream}  # 上游分支
git merge-base HEAD origin/main         # 与主干的共同祖先
git merge-base HEAD origin/master
```

若上游或主干分支不存在属于正常情况，可回退到可用分支或说明信息缺失，不得将查询失败当作结论依据。

## PR 附件规范

PR 正文需要截图、终端图片作为证据时，遵循仓库内 `docs/reference/github-attachment-upload.md` 规范。仅允许使用 GitHub 官方附件上传通道，不得将临时图片提交到仓库、不得用 GitHub Releases 存放 PR 证据、不得使用外部图床。禁止记录浏览器 Cookie、CSRF 令牌、S3 表单字段等敏感信息。

## 提交模式

仅提交用户指定的变更，保留无关的未提交内容。

1. 从近期提交历史中识别提交信息风格，沿用仓库主流的格式、语言和大小写；仓库无统一约定时，不默认使用约定式提交规范。

2. 完整审查全部差异内容，而非仅查看文件名，区分用户要求提交的内容与无关修改。

3. 按功能、模块、可回滚性划分原子提交组，功能实现与对应测试放在同一提交。

4. 无关内容优先拆分为多个提交；仅当变更属于不可分割的同一功能，或用户明确要求单次提交时，才合并为一个提交。

5. 按文件路径或代码块粒度暂存，确保每个提交仅包含对应原子组的内容。

6. 每次提交前，通过 `git diff --staged --stat` 验证暂存内容准确。

7. 按识别的风格撰写提交信息，提交后验证结果。

**分组规则**：

- 不同功能、模块、生成产物、配置、文档、纯测试修改需拆分，除非内容不可分割

- 生成文件需与对应源码变更一同提交，避免仓库状态不一致

- 不得将有问题或无关的变更隐藏在大范围提交中

**最终输出**：列出提交哈希、提交信息，以及剩余未提交的文件。

## 变基模式

历史改写会影响协作，需谨慎执行。

- 除非用户明确指定，否则不得对 `main`、`master`、`dev`、发布分支等受保护分支执行变基或历史改写。

- 若提交可能已推送，强制推送前必须询问用户；强制推送使用 `--force-with-lease`，禁止使用普通 `--force`。

- 工作区存在未提交内容时，变基前先妥善保存；解决冲突后再弹出暂存，不得盲目覆盖内容。

- fixup 操作优先使用 `git commit --fixup=<哈希>` \+ `GIT_SEQUENCE_EDITOR=: git rebase -i --autosquash <基准分支>` 流程。

- 遇到冲突时，结合代码意图解决，不得盲目使用 ours/theirs 策略。

- 变基出错时优先执行 `git rebase --abort`；需要恢复历史时再说明 reflog 方案。

改写完成后，运行相关测试（至少执行最简冒烟检查），并展示从基准到 HEAD 的新分支提交日志。

## 历史查询模式

根据问题选择对应工具：

- `git log -S "文本"`：定位精确字符串出现次数发生变化的提交

- `git log -G "正则"`：定位差异内容匹配正则表达式的提交

- `git blame -L 起始行,结束行 -- 文件路径`：追溯指定行的最后修改者

- `git log --follow -- 文件路径`：查询文件跨重命名的完整历史

- `git show <提交哈希>`：查看具体提交的详细变更

- `git bisect`：存在明确的通过 / 失败判定条件时，定位第一个引入问题的提交

- `git reflog`：恢复或解释本地近期的分支操作历史

回答中必须附上对应命令的证据：提交哈希、标题、文件路径，以及相关行号或差异上下文。证据不明确时，需说明尚未证实的内容。

## 安全检查

**任何 Git 历史写入操作前，必须确认：**

- 明确当前分支名称

- 已统计所有未提交的工作区变更

- 已知晓上游 / 推送状态，或明确说明状态未知

- 操作内容与用户需求完全匹配

- 具备回滚方案（如 `rebase --abort`、reflog 哈希、保留完整工作区）

**操作完成前：**

- 对变更或历史操作执行最相关的验证

- 说明已执行通过的命令，以及无法执行的命令

- 明确告知最终的工作区状态
