# 智能体工作流规则

这些规则定义了基于多智能体协作的任务执行工作流，适用于所有需要规划与执行的开发任务。

## 任务路由

Sisyphus 作为主控智能体，根据任务规模分类和路由：

- **琐事（Trivial）**：单步操作、简单问答——Sisyphus 直接执行，无需委派。
- **小任务（Small）**：2-5 步、单模块范围——委派给 Sisyphus-Junior 直接执行。
- **中大型任务（Medium/Large）**：5+ 步、跨模块、需架构决策——进入规划流程：Prometheus 规划 → Atlas 执行。

## 规划流程

中大型任务的完整规划路径：

1. **Prometheus 规划**：Prometheus 首先深度探索代码库，进行需求澄清和预分析，收集上下文，生成一份决策完备的计划文档。
2. **用户确认**：计划提交用户审核，用户需明确批准后方可进入下一阶段。
3. **Momus 审核**：计划生成后，Prometheus 自动调用 Momus 审核计划的完整性和可执行性。
4. **Atlas 执行**：审核通过后，Atlas 接管执行阶段。

## 计划审核

Momus 负责审核 Prometheus 生成的计划：

- 触发时机：Prometheus 生成计划后自动调用 Momus。
- 审核标准：检查引用文件是否存在、任务是否可执行、是否存在关键阻塞点、QA 场景是否可执行。
- 默认批准通过；发现问题时，最多提出 3 个阻塞性问题。
- 阻塞性问题必须由 Prometheus 修正后重新提交审核。

## 计划与跟踪文件位置

- 计划文件：`.opencode/tasks/<任务名称>/plan.md`
- 跟踪文件：`.opencode/tasks/<任务名称>/boulder.json`
- `plan.md` 是 Atlas 执行的唯一依据，包含步骤定义、依赖关系和验收标准。
- `boulder.json` 记录执行进度、各步骤状态和累积发现。

## 执行流程

Atlas 接管后的标准执行路径：

1. **读取计划**：解析 `plan.md`，构建步骤依赖图。
2. **并行执行**：无依赖的步骤并行委派给 Sisyphus-Junior 子代理。
3. **独立验证**：每个步骤完成后由独立验证检查结果质量。
4. **累积发现**：执行过程中产生的洞察、问题和解决方案统一记录。
5. **更新跟踪**：实时更新 `boulder.json`，反映当前进度和各步骤状态。

## 子代理约束

各子代理的能力边界：

- **Sisyphus-Junior**：聚焦执行器，负责具体编码和实现。**不可重新委派任务**给其他代理。
- **Oracle**：只读代码审查，负责目标验证、代码质量检查和安全性分析。
- **Librarian**：只读知识检索，负责文档查询、代码搜索和外部资料收集。
- **Multimodal-Looker**：只读媒体分析，负责图片、PDF 等非文本文件的视觉信息提取。
- **Momus**：只读计划审核，负责检查计划的完整性和可行性，不参与执行。

## 启动执行

用户批准计划后，使用以下命令启动 Atlas 执行：

```
/start-work
```

Atlas 会自动定位 `plan.md` 和 `boulder.json`，按计划编排执行工作流。

## 会话连续性

保持子智能体会话上下文的方法：

- 每个子代理任务使用 `task_id`（格式：`ses_...`）标识唯一会话。
- 后续对同一子代理的委派传入 `session_id` 参数，**延续**已有会话上下文。
- 避免每次委派创建全新会话，减少重复上下文加载和探索开销。
- 会话在任务完成后自动结束，无需手动清理。
