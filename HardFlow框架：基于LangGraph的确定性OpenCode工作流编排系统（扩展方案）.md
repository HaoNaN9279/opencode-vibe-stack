# HardFlow框架：基于LangGraph的确定性OpenCode工作流编排系统（扩展方案）

## 核心架构改进与设计原则

### 关键设计原则（必须严格遵守）

1. **控制与执行绝对分离**：工作流的**流转逻辑 100% 由代码定义**，AI 仅负责两个边界：用户自然语言交互和单个节点内的任务执行

2. **状态单向流动**：所有状态变更只能通过工作流节点的输出产生，禁止 AI 直接修改全局状态

3. **可观测性优先**：每个节点的执行状态、输入输出、耗时都必须可追踪、可审计

4. **失败可恢复**：任何节点失败都能从断点重试，支持手动干预和回滚

### 整体架构分层

```Plain Text

├─────────────────────────────────┤

```

## 关键组件技术原理详解

### 1\. 统一管理 Agent（用户交互层）

**核心职责**：作为用户与系统的唯一接口，不参与任何工作流内部执行逻辑

**技术原理**：

- **有限能力 Agent 设计**：仅赋予三个明确的工具调用权限：

    1. `list\_workflows\(\)`：查询所有可用工作流及其描述和参数

    2. `start\_workflow\(workflow\_id, params\)`：启动指定工作流

    3. `get\_workflow\_status\(execution\_id\)`：查询工作流执行状态

- **意图识别与参数提取**：使用结构化输出（JSON Schema）将用户自然语言请求转换为工作流 ID 和参数对象

- **状态报告生成**：订阅事件总线的状态变更事件，将结构化的状态信息转换为自然语言报告

- **会话上下文管理**：维护用户会话与工作流执行 ID 的映射关系，支持多工作流并发跟踪

**关键改进点**：

- 管理 Agent**永远不会直接执行任务**，也不会修改工作流定义

- 所有用户输入都必须经过验证，确保参数符合目标工作流的 Schema

- 当用户需求无法匹配任何预定义工作流时，明确告知用户可用的工作流列表，而不是尝试动态生成

### 2\. 工作流注册表与调度器

**核心职责**：管理所有可用工作流的元数据，负责工作流实例的生命周期管理

**技术原理**：

- **工作流元数据模型**：每个工作流注册时必须提供：

    - 唯一 ID 和人类可读名称

    - 详细的功能描述（用于管理 Agent 的意图匹配）

    - 输入参数 Schema（JSON Schema 格式）

    - 输出参数 Schema

    - 作者和版本信息

- **工作流工厂模式**：工作流定义为可实例化的类，每次执行创建独立的 LangGraph 实例

- **执行 ID 生成**：每个工作流实例分配全局唯一的执行 ID，作为状态追踪的主键

- **并发控制**：基于令牌桶算法限制同时运行的工作流数量，防止 OpenCode API 过载

**关键改进点**：

- 支持工作流的热加载和热更新，无需重启系统

- 支持工作流的版本控制，不同版本可以并行运行

- 支持工作流的依赖管理，一个工作流可以作为另一个工作流的子节点

### 3\. LangGraph 执行引擎与 OpenCode 节点适配器

**核心职责**：执行预定义的工作流逻辑，每个节点对应一次 OpenCode 调用

**技术原理**：

- **标准节点接口**：所有 OpenCode 节点实现统一的接口：

    ```Plain Text
    interface HardFlowNode {
      id: string;
      name: string;
      description: string;
      agent: string;  // 指定使用的OMO Agent
      input_schema: JSONSchema;
      output_schema: JSONSchema;
      timeout: number;
      max_retries: number;
    }
    ```

- **OpenCode 调用封装**：

    - 使用 OpenCode 的 HTTP API 而非 CLI 调用，提高性能和可靠性

    - 支持异步调用和轮询状态，避免长时间阻塞

    - 自动处理 API 速率限制和重试

- **节点状态模型**：每个节点执行后返回标准化的状态对象：

    ```Plain Text
    interface NodeExecutionState {
      node_id: string;
      execution_id: string;
      status: "pending" | "running" | "completed" | "failed" | "skipped";
      start_time: timestamp;
      end_time: timestamp;
      input: any;
      output: any;
      error: string | null;
      logs: string[];
    }
    ```

- **条件分支与循环**：完全使用 LangGraph 的原生条件边和循环机制实现，所有分支逻辑由代码定义

**关键改进点**：

- 支持节点级别的超时和重试配置

- 支持节点的跳过和手动执行

- 支持并行节点执行，提高工作流效率

- 支持节点的输入输出转换，实现不同节点之间的数据格式兼容

### 4\. 全局状态管理系统

**核心职责**：维护工作流执行过程中的所有状态，支持断点续跑和历史查询

**技术原理**：

- **分层状态设计**：

    1. **工作流实例状态**：全局状态，包含所有节点的执行结果和元数据

    2. **节点执行状态**：每个节点的独立状态，包含输入输出和执行信息

    3. **用户会话状态**：用户与系统交互的上下文信息

- **状态持久化**：

    - 使用键值数据库（如 Redis）存储活跃工作流的状态

    - 使用关系型数据库（如 PostgreSQL）存储历史执行记录

    - 每次节点执行完成后自动持久化状态

- **状态不可变性**：所有状态变更都是追加式的，保留完整的历史版本

- **状态恢复机制**：系统重启后可以从数据库中恢复所有未完成的工作流实例

**关键改进点**：

- 支持状态的快照和回滚

- 支持状态的导出和导入

- 支持状态的查询和过滤，便于调试和审计

- 实现状态的 TTL 自动清理，避免数据库膨胀

### 5\. 事件总线与通知系统

**核心职责**：实现组件间的解耦通信，将工作流状态变更实时推送给用户

**技术原理**：

- **发布 \- 订阅模式**：所有组件通过事件总线进行通信，不直接依赖

- **标准化事件类型**：

    - `WorkflowStartedEvent`：工作流启动

    - `NodeStartedEvent`：节点开始执行

    - `NodeCompletedEvent`：节点执行完成

    - `NodeFailedEvent`：节点执行失败

    - `WorkflowCompletedEvent`：工作流完成

    - `WorkflowFailedEvent`：工作流失败

- **实时通知机制**：

    - 管理 Agent 订阅事件总线，将事件转换为自然语言报告推送给用户

    - 支持多种通知渠道：聊天界面、邮件、Webhook 等

- **事件持久化**：所有事件都被持久化到数据库，用于后续的审计和分析

**关键改进点**：

- 支持事件的过滤和路由，用户可以自定义关注的事件类型

- 支持事件的重放，便于调试和问题排查

- 实现事件的幂等性处理，防止重复通知

### 6\. 错误处理与恢复机制

**核心职责**：确保系统在出现错误时能够优雅处理，最大限度减少人工干预

**技术原理**：

- **分级错误处理**：

    1. **节点级错误**：自动重试指定次数，重试失败后标记为失败

    2. **工作流级错误**：根据错误类型决定是终止工作流还是跳过该节点

    3. **系统级错误**：触发告警，通知管理员进行处理

- **断点续跑**：工作流失败后，可以从失败的节点重新开始执行

- **手动干预接口**：

    - 支持手动修改节点的输入参数

    - 支持手动标记节点为成功或失败

    - 支持手动跳过某个节点

- **告警机制**：当工作流失败次数超过阈值或出现系统级错误时，自动发送告警通知

## 方案优势总结

1. **绝对确定性**：工作流的每一步流转都由代码精确控制，完全消除了 AI 决策的随机性

2. **强大的执行能力**：充分复用 OpenCode 和 OMO 的专业 Agent 能力，专注于任务执行

3. **优秀的用户体验**：通过统一管理 Agent 提供自然语言交互，隐藏底层技术细节

4. **高度可观测性**：完整的状态追踪和事件通知，让用户随时了解工作流执行进度

5. **健壮的错误处理**：完善的重试、回滚和手动干预机制，确保系统的可靠性

6. **良好的可扩展性**：模块化的架构设计，便于添加新的工作流和功能

## 适用场景与局限性

**适用场景**：

- 重复性高、流程固定的开发任务（如 TDD 开发、代码审查、部署等）

- 对执行结果的一致性和可靠性要求高的场景

- 需要严格按照规定流程执行的任务

- 多 Agent 协作的复杂任务

**局限性**：

- 不适合完全未知、流程不固定的探索性任务

- 工作流的定义和维护需要一定的开发成本

- 无法处理超出预定义工作流范围的需求
