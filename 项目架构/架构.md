# Vibe Stack 架构设计

## 1. 项目定位

**Vibe Stack** 是面向 OpenCode 的分层 AI 智能体配置管理工具。管理不同开发领域的配置集合——规则、技能、智能体定义、命令和 MCP 服务器，按需组合加载。

### 核心约束

1. **仅依赖 OpenCode 原生能力** — 不依赖 OhMyOpenAgent (OMO) 或其他插件
2. **命令触发更新** — 源配置变更后不会自动传播到项目，需运行 `vibe-stack sync`
3. **不破坏项目自定义内容** — `.opencode/` 下项目自己的文件永不被触碰

## 4. 核心概念

### 4.1 领域（Domain）—— 最小配置单元

每个领域通过 `domain.json` 自描述，放置在任意深度的目录中：

```json
{
  "$schema": "https://vibe-stack.dev/domain.schema.json",
  "name": "data-forge",
  "description": "Data transformation and AI training data pipeline",
  "version": "1.0.0",
  "tags": ["data", "ai", "training"],
  "dependencies": []
}
```

**领域键** = 目录相对 `domains/` 的路径（正斜杠分隔）：

```
domains/dcc/blender/domain.json          → 键: dcc/blender
domains/dcc/subcategory/deep/domain.json → 键: dcc/subcategory/deep
domains/ai/data-forge/domain.json        → 键: ai/data-forge
```

**发现算法**：`domains/**/domain.json` 递归匹配。不依赖深度限制。

### 4.2 命名空间（Namespace）—— 防冲突隔离

每个领域激活后在 `.opencode/{type}/` 下拥有**独立子目录**：

```
领域键: dcc/blender
命名空间: dcc_blender                    (/ → _)

.opencode/rules/dcc_blender/            ← 独立目录，不与其他领域碰撞
.opencode/agents/dcc_blender/
.opencode/commands/dcc_blender/
.opencode/mcp/dcc_blender/
.opencode/skills/dcc_blender/
```

命名规则：`领域键.replace("/", "_")`

这种设计的优势：
- **文件名可读**：`rules/dcc_blender/blender.md`，保持原始文件名
- **删除干净**：直接 `rm -rf .opencode/rules/dcc_blender/`
- **项目内容不受影响**：`.opencode/rules/project-rule.md` 永不被触碰
- **OpenCode 兼容**：`instructions` 支持 glob `.opencode/rules/dcc_blender/*.md`

### 4.3 技能与工具（Skills + Tools）—— 集中管理

**所有工具集中在仓库根目录的 `tools/` 下**，不随 skill 拷贝到项目。每个 skill 的 `SKILL.md` 通过引用自动生成的规则文件来定位工具路径：

```
仓库根目录/
├── tools/                               ← 所有工具集中在这里
│   ├── markitdown/
│   │   ├── pyproject.toml
│   │   └── src/
│   └── data-forge/
│       ├── pyproject.toml
│       └── src/
├── core/skills/markitdown/
│   └── SKILL.md                         ← 引用 tools/markitdown/
│
~/.config/opencode/rules/
└── vibe-stack-tools.md                  ← sync 时自动生成，含绝对路径
```

**工作方式**：

1. `vibe-stack sync` 运行时，`engine.py` 中的 `_sync_tools_rule()` 扫描 `tools/` 下所有含 `pyproject.toml` 的目录
2. 提取每个工具的名称和入口命令，写入 `~/.config/opencode/rules/vibe-stack-tools.md`
3. AI 智能体启动时自动加载该规则文件，通过其中的**绝对路径**定位并执行工具
4. skill 的 `SKILL.md` 不再捆绑工具代码，改为指引 AI 读取 `vibe-stack-tools.md` 获取路径

```
# 工具代码在 SKILL.md 中的引用方式
工具位于 Vibe Stack 仓库的 `tools/markitdown/`，具体路径见
`~/.config/opencode/rules/vibe-stack-tools.md`。
使用：uv run --project <路径> markitdown <参数>
```

**这种方式的好处**：
- **避免冗余拷贝**：工具代码无需随每个领域激活而复制到项目 `.opencode/`
- **单一真相来源**：工具只存放在 `tools/` 下一份，修改后 sync 即可生效
- **AI 可发现**：规则文件自动加载，AI 智能体始终知道工具的准确路径
- **按需安装依赖**：AI 在首次使用工具时执行 `uv sync` 安装依赖

### 4.4 所有权（Ownership）—— 项目内容保护

`.opencode/.vibe-stack-state.json` 记录 vibe-stack 拥有的全部文件、目录和配置项：

```json
{
  "version": 2,
  "domains": {
    "dcc/blender": {
      "namespace": "dcc_blender",
      "activated_at": "2026-07-10T14:00:00Z",
      "files": [
        "rules/dcc_blender/blender.md",
        "agents/dcc_blender/blender-agent.md"
      ],
      "directories": [
        "skills/dcc_blender/blender-tool/",
        "mcp/dcc_blender/blender-mcp/"
      ],
      "opencode_entries": {
        "instructions": [".opencode/rules/dcc_blender/*.md"],
        "skills.paths": [".opencode/skills/dcc_blender"],
        "mcpServers": ["vibe:dcc_blender_blender-mcp"]
      }
    }
  }
}
```

**所有权规则**（三条铁律）：
1. **只删除/修改状态文件中记录的内容** — 停用时精确删除
2. **只更新 `vibe:` 前缀的配置项** — 在 opencode.json 中只操作 `vibe:` 开头的 mcpServers key
3. **永不修改用户自定义内容** — `.opencode/rules/my-project-rule.md`、自建的 mcpServers、自加的 instructions 条目，全部保留

## 5. 模块职责

### 5.1 Model 层 — 纯数据模型

```python
# model/domain.py
@dataclass
class DomainMeta:
    """领域元数据，来自 domain.json"""
    name: str
    description: str
    version: str
    tags: list[str]
    dependencies: list[str]

@dataclass
class DomainConfig:
    """领域的完整配置内容"""
    meta: DomainMeta
    domain_key: str                # 如 "dcc/blender"
    namespace: str                 # 如 "dcc_blender"
    domain_root: Path              # 领域在仓库中的根路径
    rules: list[Path]              # rules/*.md 文件列表
    agents: list[Path]             # agents/*.md 文件列表
    commands: list[Path]           # commands/*.md 文件列表
    mcp_files: list[Path]          # mcp/*.json 文件列表
    mcp_dirs: list[Path]           # mcp/{name}/ 辅助目录列表
    skills: list[Path]             # skills/*/ 目录列表（每个含 SKILL.md）
```

```python
# model/state.py
@dataclass
class ActivationState:
    """.vibe-stack-state.json 的数据模型"""
    version: int
    domains: dict[str, DomainState]     # domain_key → DomainState

@dataclass
class DomainState:
    """单个领域的激活状态"""
    domain_key: str
    namespace: str
    activated_at: str
    files: list[str]                    # 相对 .opencode/ 的文件路径
```

### 5.2 Loader 层

| 模块 | 函数 | 职责 |
|------|------|------|
| `domain_loader.py` | `discover_domains()` | 递归扫描 `domains/` 下所有含 `domain.json` 的目录 |
| `domain_loader.py` | `load_domain()` | 读取指定领域的完整配置（meta + 扫描各类型目录） |
| `core_loader.py` | `load_core()` | 加载 `core/` 下的所有配置 |
| `stack_loader.py` | `load_stack()` | 读取 stack JSON，展开为领域键列表 |

领域发现：

```python
def discover_domains(vibe_home: Path) -> list[tuple[str, DomainMeta, Path]]:
    """递归扫描 domains/ 下所有含 domain.json 的目录。
    
    返回: [(domain_key, meta, domain_root), ...]
    排序保证确定性。
    """
    result = []
    for domain_json in sorted((vibe_home / "domains").rglob("domain.json")):
        domain_root = domain_json.parent
        relative = domain_root.relative_to(vibe_home / "domains")
        domain_key = str(relative.as_posix())   # 如 "dcc/sub/deep"
        meta = _parse_domain_json(domain_json)
        result.append((domain_key, meta, domain_root))
    return result
```

#### mcp_resolver.py — MCP 路径解析

MCP JSON 声明中的命令支持占位符：

```json
{
  "mcpServers": {
    "blender-mcp": {
      "type": "local",
      "command": ["${MCP_PATH_BLENDER_MCP}", "--port=8080"],
      "enabled": true
    }
  }
}
```

解析优先级：
1. 环境变量 `MCP_PATH_BLENDER_MCP`
2. 环境变量 `MCP_PATH`（通用回退）
3. `~/.config/opencode/vibe-mcp-paths.json`
4. 原样使用（依赖 PATH）


| `activate <d...>` | 激活领域 | 只追加，不删除任何内容 |
| `deactivate <d...>` | 停用领域 | 只删除状态文件记录的条目 |
| `use-stack <name>` | 激活预设堆栈 | 同 activate |
| `sync` | 刷新 Core + 所有激活领域 | 只更新状态文件记录的内容 |

`sync` 命令完整流程：

```
vibe-stack sync
  │
  ├── Part A: Core Sync  ────────────────────────────
  │   ├── 拷贝 core/rules/* → ~/.config/opencode/rules/
  │   ├── 拷贝 core/skills/* → ~/.config/opencode/skills/
  │   ├── 拷贝 core/agents/* → ~/.config/opencode/agents/
  │   ├── 拷贝 core/commands/* → ~/.config/opencode/commands/
  │   ├── 拷贝 core/mcp/* → ~/.config/opencode/mcp/
  │   ├── 写入 ~/.config/opencode/opencode.json
  │   └── 生成 ~/.config/opencode/rules/vibe-stack-tools.md  ← 工具索引
  │
  └── Part B: Domain Sync ──────────────────────────
      ├── 读取 .vibe-stack-state.json
      ├── 对每个激活的领域:
      │   ├── 重新加载领域配置（从源）
      │   ├── 增量拷贝文件到 .opencode/{namespace}/
      │   └── 删除源端已不存在的文件
      └── 重写 .opencode/opencode.json
```

## 6. 关键设计决策

### 6.1 文件拷贝替代符号链接

**决策**：领域激活时将文件**拷贝**到 `.opencode/{namespace}/`，而非创建符号链接。

**理由**：
- 满足需求 ②：源文件变更需要**显式 sync** 才传播，不会自动影响项目
- 避免 Windows 符号链接/junction 的兼容性问题
- 拷贝后的文件是独立的，项目可以离线使用

### 6.2 命名空间子目录替代文件名前缀

**决策**：每个领域在 `.opencode/{type}/` 下创建独立子目录。

```
✅ 新: .opencode/rules/dcc_blender/blender.md   ← 文件名可读
❌ 旧: .opencode/rules/dcc_blender_blender.md   ← 文件名膨胀不可读
```

**理由**：
- 保持原始文件名
- 删除领域时直接删除整个目录（简单、干净）
- OpenCode 的 `instructions` 支持 glob `dcc_blender/*.md`
- OpenCode 的 `skills.paths` 支持目录路径 `dcc_blender`

### 6.3 状态文件追踪所有权

**决策**：`.opencode/.vibe-stack-state.json` 记录所有权。

**理由**（满足需求 ④）：
- 状态文件中未记录的内容永不被修改/删除
- 停用领域时精确知道要删除什么
- sync 时知道要刷新什么
- 项目专有内容完全不受影响

### 6.4 domain.json 标记领域根

**决策**：每个领域必须包含 `domain.json`，支持任意深度嵌套。

**理由**（满足需求 ⑤）：
- `domains/**/domain.json` 递归匹配发现，不限制层级
- 领域键由相对路径自动推导
- 不依赖目录结构约定，灵活扩展

### 6.5 集中管理 tools/ + 自动生成工具索引

**决策**：所有工具集中在根 `tools/` 目录，`sync` 时自动生成 `vibe-stack-tools.md` 规则文件。

**理由**：
- **避免冗余**：工具代码不随领域激活拷贝到项目，`.opencode/` 保持轻量
- **单一来源**：工具只存在一份，修改后 sync 即全局生效
- **AI 可发现**：`vibe-stack-tools.md` 被加载为规则文件，AI 智能体始终知道工具的绝对路径
- **按需安装**：工具依赖不预装，AI 首次使用时执行 `uv sync`
- **与 6.3 状态文件原则一致**：`tools/` 不属于任何领域，不进入 `.vibe-stack-state.json`

每个工具是一个独立的 Python 项目（含 `pyproject.toml`），通过 `uv run --project <path>` 执行。`_sync_tools_rule()` 引擎负责自动发现和索引生成。

### 6.6 仅依赖 OpenCode

**决策**：只处理 `opencode.json`，不处理 `oh-my-openagent.jsonc`。

**理由**（满足需求 ①）：
- 移除所有 OMO 相关代码（`agent_definitions`、`keyword_detector`、模式注入等）
- 配置目标只有一个：`opencode.json`
- OpenCode 原生支持的配置项：`instructions`、`skills.paths`、`mcpServers`、`agents`、`commands`

### 6.7 MCP 路径解析简化

**决策**：废弃独立的 Registry 子系统，使用环境变量 + 轻量配置。

解析优先级（从高到低）：
1. `MCP_PATH_{SERVER_NAME}` 环境变量
2. `MCP_PATH` 环境变量（通用回退）
3. `~/.config/opencode/vibe-mcp-paths.json`
4. 命令字符串原样（依赖 PATH）

```json
// ~/.config/opencode/vibe-mcp-paths.json
{ "codebase-memory": "/usr/local/bin/codebase-memory-mcp" }
```

## 7. 与原架构的关键差异

| 维度 | 原架构 | 新架构 |
|------|--------|--------|
| **依赖** | OpenCode + OMO | **仅 OpenCode** |
| **插件依赖** | 需要 OMO 插件 | **零插件依赖** |
| **文件传播** | 符号链接（自动同步） | **文件拷贝（sync 命令触发）** |
| **tools/ 目录** | 独立存在 | **集中管理到根 tools/ + sync 自动生成索引** |
| **项目专有内容** | 无保护机制 | **状态文件追踪所有权，永不触碰** |
| **领域嵌套** | 固定 2 级（cat/name） | **任意深度（通过 domain.json 发现）** |
| **命名冲突** | Per-item 文件名前缀 | **命名空间子目录（隔离清晰）** |
| **MCP 管理** | Registry 子系统 | **环境变量 + 轻量映射** |
| **状态文件** | `.vibe-stack-active.json` + 链接映射 | **`.vibe-stack-state.json`（纯所有权记录）** |
| **配置写入** | 文本级 JSON 操作 | **dict 级合并操作** |
| **配置目标** | opencode.json + oh-my-openagent.jsonc | **仅 opencode.json** |
| **命令** | activate, deactivate, list, status, use-stack, core-update | **activate, deactivate, list, status, use-stack, sync, info** |
| **安装步骤** | 8 步（含 symlink + registry + CLI wrapper） | **3 步（clone → deps → sync）** |

## 9. 安装与使用

### 安装（3 步）

```bash
# 1. 克隆
git clone https://github.com/your-org/opencode-vibe-stack.git ~/.opencode-vibe-stack

# 2. 安装 Python 依赖
cd ~/.opencode-vibe-stack && uv sync

# 3. 初次同步（Core → ~/.config/opencode/）
uv run vibe-stack sync
```

### 日常使用

```bash
# 在项目中激活领域
cd my-project
vibe-stack activate dcc/blender ai/data-forge

# 查看状态
vibe-stack status

# 查看领域详情
vibe-stack info dcc/blender

# 源配置更新后，同步到项目
vibe-stack sync

# 停用领域
vibe-stack deactivate dcc/blender
```
